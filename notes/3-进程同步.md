# 进程同步

## 进程同步的基本概念

### 进程间的三种关系

进程间的相互关系主要有三种：

- `同步`
- `互斥`
- `通信`

#### 同步

同步强调的是`合作进程之间的先后次序协调`。

- 若干进程为了完成共同任务，需要按一定顺序推进。
- 一个进程执行某个操作之前，可能必须等待另一个进程先完成某个操作。

例如：

- 教师出卷 -> 学生考试 -> 教师改卷 -> 学生订正

这个过程中的每一步都有明确的先后依赖，这就是同步关系。

#### 互斥

互斥强调的是`共享独占资源时的排他访问`。

- 当多个进程共享某个独占性资源时，必须保证任一时刻只有一个进程能访问它。
- 否则就会发生数据错误、资源冲突或结果不一致。

#### 二者关系

课件特别强调：`互斥关系也是同步关系，是一种特殊的同步。`

也就是说：

- 同步更广，强调协作与时序
- 互斥更具体，强调访问共享独占资源时不能同时进入

### 临界资源

#### 定义

`临界资源（Critical Resource）：一次只允许一个进程访问的资源，进程必须通过`互斥方式来共享临界资源。

对临界资源的访问可以在逻辑上分为以下四部分：

```c
while (true) {
    entry_section;      // 进入区：检查临界资源能否访问，并申请进入临界区
    critical_section;   // 临界区：真正访问临界资源的代码
    exit_section;       // 退出区：退出临界区并释放资源
    remainder_section;  // 剩余区：其余与临界资源无关的代码
}
```

### 临界资源的原则

访问临界资源应满足的原则：

- `空闲让进`：当互斥区空闲时，任何有权进入的进程都可以进入。
- `忙则等待`：若已有进程在互斥区内，其他进程必须等待。
- `有限等待`：每个进入请求都应在有限时间内得到满足，不能无限期等待。
- `让权等待`：处于等待状态的进程应放弃占用 CPU，让其他进程运行。

其中，`让权等待` 是避免“忙等”浪费处理机时间的关键。

## 硬件同步机制

### 硬件同步机制的三种典型方法

课件给出三种典型方式：

- `关中断`
- `Test-and-Set 指令`
- `Swap 指令`

### 关中断

操作：进程进入临界区后关闭中断，离开临界区前再打开中断。

> [!NOTE]
>
> 这样做的目的，是让当前 CPU 在执行临界区代码时不被时钟中断打断，也就不会发生进程切换。

优点：

- 简单高效

缺点：

- 不适合多处理机
- 只适用于操作系统内核进程，不适用于用户进程 (这组指令如果让用户随意使用会很危险)

![int](/Users/yzhbradoodrrpurp/Desktop/OS/notes/resources/int.png)

### Test-and-Set 指令

操作：它会把以下两个动作合并成一个不可分割的整体：

- `测试（Test）`
  检查共享变量当前的值
- `设置（Set）`
  把该变量原子地改成新值

正因为这两个动作不可分割，所以其他进程无法在中间插入，从而避免了竞态条件。

![testandset](/Users/yzhbradoodrrpurp/Desktop/OS/notes/resources/testandset.png)

优点：

- 能有效实现互斥

缺点：

- 不符合上一节中的`让权等待`原则

### Swap 指令

操作：

- 为每个临界资源设置一个全局布尔变量 `lock`
- 初始值设为 `false`
- 进程进入临界区前准备一个局部变量 `old = true`
- 执行 `swap(old, lock)`

交换后：

- 如果 `old == false`，说明交换前 `lock` 是空闲的，当前进程获得锁，可以进入临界区
- 如果 `old != false`，说明锁已被其他进程占用，需要继续等待

退出临界区时，同样把 `lock` 置回 `false`。

![swap](/Users/yzhbradoodrrpurp/Desktop/OS/notes/resources/swap.png)

优点：

- 能有效实现互斥

缺点：

- 不符合上一节中的`让权等待`原则

## 信号量机制实现互斥

### 为什么要引入信号量机制

前面的硬件同步机制虽然能够实现互斥，但存在一个明显问题：

- 当资源被占用时，其他进程只能不断测试，这会形成`忙等`
- 不符合`让权等待`原则

因此，需要一种更适合进程同步的工具来协调并发进程，这就是`信号量机制`。

### 什么是信号量机制

信号量：其实就是一个变量 (可以是一个整数，也可以是更复杂的记录型变量)，可以用一个信号量来表示系统中某种资源的数量，比如系统中只有一台打印机，就可以设置一个初值为1的信号量。

> [!NOTE]
>
> 用户进程可以通过使用操作系统提供的一对原语来对信号量进行操作。
>
> 原语：一种特殊的程序段，其执行只能一气呵成，不能被中断。

### P 操作和 V 操作

**对信号量的访问不能像普通变量那样随意读写，只能通过两个标准原语操作**：

- `wait(S)`：也叫 `P(S)`，检测是否还能使用某类资源；如果能用，就获得资源；如果不能，就等待。
- `signal(S)`：也叫 `V(S)`，表示进程用完资源后释放资源，并通知等待的进程。

### 整型信号量

定义：用一个整型数变量作为信号量，表示某类资源的可用数量。

它的典型形式是：

```c++
int S = 1; // 初始化整型信号量S，表示当前系统中可用的打印机数

// wait原语，相当于进入区
// 如果资源不够，则一直等待；如果资源够，则占用一个资源
void wait(int S) { 
  while (S <= 0);
    S -= 1;
}

// signal原语，相当于退出区
// 使用完资源后，在退出区释放资源
void signal(int S) {
  S += 1;
}
```

#### 优点

- 思想简单，容易理解
- 能表达资源数量

#### 缺点

- 当 `S <= 0` 时，进程会一直循环检测，因而会产生`忙等`，不能满足`让权等待`

### 记录型信号量

#### 为什么要改进

整型信号量的缺陷是存在**忙等**问题，因此人们又提出了记录型信号量，即用记录型数据结构表示的信号量。

```c++
typedef struct {
    int value; // 剩余资源数
    struct process *list; // 等待队列
} semaphore;
```

#### 基本操作

```c++
void wait(semaphore S) {
    S.value--;
    if (S.value < 0) 
      block(S.list); // 如果剩余资源数不够，使用block原语使进程从运行态进入阻塞态，并将进程挂到等待队列中
}

void signal(S) {
    S.value++;
    if (S.value <= 0) 
      wakeup(S.list); // 如果一个进程释放后剩余资源还是不够，则从等待队列中激活一个进程，从阻塞态变为就绪态
}
```

#### 优点

- 等待资源的进程会执行 `block`，进入阻塞队列
- 这样就不会忙等，满足`让权等待`
- 比整型信号量更符合操作系统对进程同步的要求

### AND 型信号量

#### 使用场景

当一个进程需要`同时申请多种不同资源`，并且每种资源都只占用一个，因此引入`AND 型信号量`。

#### 基本思想

在一个原语中一次性申请进程需要的多个资源：

- 要么`全部分配`
- 要么`一个都不分配`

#### 核心作用

- 防止只拿到部分资源却继续等待其他资源
- 降低死锁风险

### 信号量集

如果：

- 一个进程需要申请的资源种类很多
- 或者每类资源都要申请多个

那么逐个做 `P/V` 操作会很麻烦。

于是引入`信号量集`。

#### 基本思想

在一次操作中同时处理多个资源请求：

```text
Swait(S1, t1, d1; ...; Sn, tn, dn)
Ssignal(S1, d1; ...; Sn, dn)
```

其中：

- `t` 表示分配下限
- `d` 表示需求值

课件给了几个特殊情形：

- `Swait(S, d, d)`：每次申请 `d` 个资源，少于 `d` 就不分配
- `Swait(S, 1, 1)`：等价于互斥信号量
- `Swait(S, 1, 0)`：可以看作一个可控开关

#### 适用场景

信号量集适合：

- 同一进程申请多个资源
- 多个进程申请多个资源

## 管程

### 为什么要引入管程

之前实现进程同步、进程互斥主要是通过信号量机制，但是信号量机制存在一个问题：**编写程序困难，很容易出错造成死锁现象**。

管程就是一种机制，能够让程序员写程序时不需要关注复杂的 PV 操作，让写代码更轻松。

### 定义和基本特征

管程是一种特殊的软件模块，由这些部分组成：

- 局部于管程的共享数据结构
- 对该数据结构进行操作的过程 (函数)
- 对局部于管程的共享数据设置初始值的语句
- 管程的名字

> [!NOTE]
>
> 管程很类似于面向对象编程中的对象。

基本特征：

- 局部于管程的数据只能被局部于管程的过程所访问
- **一个进程只能通过调用管程内的过程才能进入管程访问共享数据**
- **每次仅允许一个进程在管程内执行某个内部过程**
